PendingIntent概述

pendingIntent与Intent的区别在于pendingIntent是在将来某个不确定的时刻发生，而Intetn是立即发生。要给RemoteViews设置点击事件，就必须使用pendingIntent，pendingIntent通过send和cancel方法来发送和取消特定的待定intent。

pendingIntent的匹配规则为：如果两个pendingIntent内部的intent和requestcode都相同，那么这两个pendingIntent相同。intent的匹配规则：如果两个intent的componentName和intent-filter都相同，那么这两个intent相同。extras不参与intent匹配过程。

requestCode表示pendingIntent发送方的请求码，多数情况下设0即可，但会影响到flags的效果。
flags类型：
FLAG_ONE_SHOT:当前描述的pendingintent只能被使用一次，然后他就会被自动cancel，如果后续还有相同的pendingintent，那么她们的send方法会调用失败。对于通知栏消息来说，那么同类通知只能使用一次，后续的通知单击后将无法打开。
FLAG_NO_CREATE:当前pendingintent不会主动创建，如果该pendingintent之前不存在，那么getActivity等会返回null，即pendingIntent获取失败。较少使用。
FLAG_CANCEL_CURRENT:当前描述的pendingIntent如果已经存在，那么他们都会被cancel，然后系统会创建一个新的pendingIntent。对通知栏来说，那些cancel的消息单击后将无法打开。
FLAG_UPDATE_CURRENT:当前描述的pendingIntent如果已经存在，那么它们都会被更新，即它们的intent中的extras都会被替换成最新的。

如果manager.notify(1,notification)的第一个参数是常量，那么多次调用notify只能弹出一个通知，后续通知会将之前的完全替换掉，不管pendingIntent是否匹配。如果ID不同，那么多次调用会弹出多个通知，如果pendingIntent不匹配，那么这些通知之间互不干扰，如果pendingIntent匹配，就得看flag标记位了，采用FLAG_ONE_SHOT时后续通知中的pendingIntent会和第一条通知保持完全一致，包括extras，单击任何一条通知后，剩下通知均无法再打开。FLAG_CANCEL_CURRENT时，只有最新的通知才可以打开。FLAG_UPDATE_CURRENT时，之前通知中的pendingIntent会被更新，与最新的一条通知保持一致，包括extras，并且都能打开。

RemoteViews的内部机制

支持类型：
layout:frameLayout,linearLayout,relativeLayout,gridLayout
view:button,imageButton,imageView,progressBar,textView,viewFlipper,listView,gridView,
stackView,adapterViewFlipper,viewStub,analogClock,Chronometer

常用方法：

方法 功能
setTextViewText(int viewId, CharSequence text) 设置textview的文本
setTextViewTextSize(int viewId, int units, float size) 设置textview的字体大小
setTextColor(int viewId, int color) 设置textview的字体颜色
setImageViewResource(int viewId, int srcId) 设置imageview的图片资源
setImageViewBitmap(int viewId, Bitmap bitmap) 设置imageview的图片资源
setInt(int viewId, String methodName, int value) 反射调用view对象的参数类型为int的方法
setLong(int viewId, String methodName, long value) 反射调用view对象的参数类型为long的方法
setBoolean(int viewId, String methodName, boolean value) 反射调用view对象的参数类型为boolean的方法
setOnClickPendingIntent(int viewId, PendingIntent pendingIntent) 为view添加点击事件，事件类型只能是pendingIntent

notificationManager通过binder和systemServer进程中的notificationManagerService进行通信，由此可知通知栏中布局文件是在notificationManagerService中被加载，而notificationManagerService运行在系统的systemServer中。
首先实现了parcelable接口的remoteviews会通过binder传递到systemserver进程，然后通过layoutinflater去加载remoteviews的布局文件，实际在systemserver中加载后的布局文件是普通的view，然后执行界面更新任务。set方法并不是立即执行的，要等到remoteviews被加载后才执行，当需要更新时，通过notificationManager提交更新任务。每调用一次set方法，remoteviews会添加一个action对象，这个action对象实现了parcelable接口，里面封装了一个view操作。当notificationManager提交更新时，这些action对象就会传输到远程进程中，远程通过remoteviews的apply方法遍历所有action对象并调用它们的apply方法来进行view的更新操作。

通知栏底层实现

如果A和B属于不同应用，通过资源ID可能无效，可以通过约定的资源名称来加载布局文件，再调用remoteviews的reapply方法将B中对view所做的更新操作作用的A中加载的view上。与apply区别是apply会加载布局并更新界面，reapply只会更新界面。
